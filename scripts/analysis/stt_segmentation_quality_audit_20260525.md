## 0. 결론 먼저

1. **1차 원인은 postprocess 병합 로직의 한계다.** VAD·WhisperX·diarization이 아니다.
   기존 v1 segmenter의 "이전 발화가 5초 미만일 때만 같은-화자 병합" 가드가
   `[긴 문장][그래서/된다.]` 형태의 짧은 연속 조각을 독립 row로 남긴다.
2. **단, 과분할의 절대량은 코퍼스 전체로 보면 제한적이다.** 발화의 71%가 2~10초로 정상
   범위이며, 경계의 다수(3,271쌍)는 2인 대화의 화자 전환(turn-taking)으로 후처리 병합 대상이
   아니다. 과분할은 일부 "short-heavy" 세션에 집중된다.
3. **Segmenter v2는 효과가 있으나, 사용자가 제시한 forward-only 정책은 너무 보수적이다.**
   같은-화자 병합 가능 인접쌍 257개 중 forward-only는 79개(31%)만 잡는다.
   **bidirectional 변형**을 쓰면 발화 −3.3%, 짧은 단편 −11.0%로 forward의 약 3배 효과.
4. **PII 경계 분리는 현재 데이터에서 0건** — 이론적 리스크일 뿐 관측되지 않음(v2가 안전망 보유).
5. **도메인 오인식은 실재**한다(선생님 24 vs 수석 24 공존, IT보안 용어 다수). hotwords/initial_prompt로 개선 가능.

## 1. 문제 유형별 수치 (전체 5,643 발화)

### 1-1. Duration 분포
| 버킷 | 수 | 비율 |
|---|---|---|
| 0s | 0 | 0% |
| <0.5s | 223 | 4.0% |
| <1s | 293 | 5.2% |
| <2s | 680 | 12.0% |
| 2~10s | 3,971 | 70.4% |
| >10s | 476 | 8.4% |

→ 71%가 정상 길이대. 초단기(<0.5s) 단편은 4%로 **VAD 과민이 주범이 아님**을 시사.

### 1-2. Text length 분포
| 항목 | 수 | 비율 |
|---|---|---|
| 1~2글자 | 170 | 3.0% |
| 3~5글자 | 452 | 8.0% |
| 1단어 | 511 | 9.1% |
| <5단어 | 1,600 | 28.4% |

### 1-3. 짧은 단편 (`<2초 OR <5단어`)
- 1,721 / 5,643 = **30.5%**.

### 1-4. 인접쌍 (경계 유형)
| 유형 | 수 |
|---|---|
| 화자 전환(speaker_change) | 3,271 |
| 같은 화자 gap ≤ 1.2s | 1,689 |
| 같은 화자 gap ≤ 0.8s | 1,097 |
| 같은 화자 gap ≤ 0.4s | 74 |

→ 경계의 **다수가 화자 전환**. 2인 통화의 정상 turn-taking이 대부분으로, 후처리 병합 대상 아님.

### 1-5. 문장 종결성
| 끝 형태 | 비율 |
|---|---|
| 종결어미(다/요/죠/까/습니다/네요…) | 15.0% |
| 연결어미(그래서/는데/니까…) | 8.5% |
| 조사(은/는/이/가/을/를…) | 17.9% |
| 기타 | 58.6% |

→ 종결어미로 끝나는 발화가 **15%뿐**. 다수가 문장 중간(조사·연결·기타)에서 잘림 = 과분할 신호.
(휴리스틱 기반 — `app/services/korean_sentence_ending.py`. "기타"에는 종결성 미검출 평서 종결도 포함되어 종결 비율은 실제보다 과소 추정될 수 있음.)

### 1-6. PII span 경계 분리
- straddle(앞 발화 우측 끝 + 뒤 발화 좌측 시작에 동시 PII) = **0건**.
- → 현재 데이터에서 전화번호/계좌 등 PII가 발화 경계로 절단되는 사례는 관측되지 않음.

### 1-7. 도메인 오인식 후보 (키워드 등장 빈도, 익명 카운트)
| 키워드 | 빈도 | | 키워드 | 빈도 |
|---|---|---|---|---|
| 메일 | 64 | | 보안 | 25 |
| 원격 | 36 | | 점검 | 21 |
| 수석 | 24 | | 서버 | 21 |
| 선생님 | 24 | | 호스트 | 17 |
| 기준 | 14 | | 로그 | 13 |
| DLP | 11 | | 백업 | 11 |
| 네트워크 | 10 | | 계정/포트 | 5/4 |

→ **선생님(24)과 수석(24) 공존** = "수석님→선생님" 오인식 가설과 정합. IT보안 도메인 용어가 코퍼스
전반에 분포. 현재 `HOTWORDS`/`INITIAL_PROMPT`는 고유명사 일부만(`data/hotword_db.json`) 다룸.

### 1-8. 같은-화자 gap≤0.8 병합 차단 사유 (1,097쌍)
| 사유 | 수 | 해석 |
|---|---|---|
| `forward_mergeable` (좌측 짧음, 미종결) | 79 | forward 정책이 잡는 유일 그룹 |
| `missed_bidir_incomplete_left` (긴 미완결 좌측 + 짧은 우측) | 178 | **forward가 놓침. bidirectional이 잡음** |
| `missed_complete_left(keep)` | 24 | 완결 문장 뒤 조각 — 병합 안 하는 게 맞음 |
| `both_long` | 808 | 둘 다 정상 길이 — 병합 대상 아님 |
| `blocked_by_left_ending` | 8 | 짧지만 종결 — 문장 경계 보존 |

→ 후처리로 고칠 수 있는 같은-화자 과분할 = **79 + 178 = 257쌍**.

## 2. 근본 원인 판정

| 후보 | 판정 | 근거 |
|---|---|---|
| VAD 과민 | ✗ 아님 | <0.5s 단편 4%에 불과. silence threshold가 주범이면 훨씬 높아야 함 |
| WhisperX word timestamp | ✗ 아님 | word 정렬 자체 문제 징후 없음. 분포는 정상 길이 중심 |
| diarization 과분할 | △ 보류 | 화자 전환 3,271쌍이 지배적이나 2인 통화에선 정상. **라벨 없이는 과분할 여부 확정 불가** → 별도 GT 검증 필요 |
| **postprocess 부재/한계** | **✓ 1차 원인** | v1의 5초 가드가 257개 같은-화자 병합 가능 쌍을 분리 유지. 종결어미 인식 전무로 문장 중간에서 끊김(종결 15%뿐) |

## 3. Segmenter v2 적용 시 감소율 추정 (메모리 dry-run, DB 미반영)

| 모드 | 발화 수 | 발화 감소 | 짧은 단편 | 단편 감소 | 종결어미 끝 |
|---|---|---|---|---|---|
| (현행 v1) | 5,643 | — | 1,721 (30.5%) | — | 15.0% |
| **v2 forward** (명세 정책) | 5,580 | **−1.1%** | 1,656 | **−3.8%** | 15.2% |
| **v2 bidirectional** (권장) | 5,456 | **−3.3%** | 1,532 | **−11.0%** | 15.5% |

- forward-only는 257쌍 중 79쌍만 잡아 효과가 작다.
- **bidirectional이 약 3배** — 같은-화자 과분할의 대부분을 해소.
- 코퍼스 전체 감소율이 한 자릿수인 이유: 발화 다수가 이미 정상 길이이고, 경계의 다수가
  병합 대상 아닌 화자 전환이기 때문. 과분할이 심한 개별 세션에서는 체감 효과가 더 크다.

> 검증: `merge_v2` 13 테스트 + v1 회귀 21 테스트 = 36 passed. 입력 불변(immutability) 테스트 포함.

## 4. hotwords / 도메인 사전 개선 목록

`INITIAL_PROMPT`에 IT보안 통화 도메인 문맥을 주고, 고빈도 오인식어를 `HOTWORDS`로 보강 권장.

- **오인식 교정 1순위**: 수석(님) ↔ 선생님 (24/24 공존 — 가장 강한 신호)
- **도메인 용어 hotword 후보**: DLP, 원격, 점검, 호스트(명), 네트워크, 방화벽, 백업, 포트, 계정, 로그, 보안, 서버, 메일
- **initial_prompt 개선안**: 현재 "이것은 한국어 통화 녹음입니다." → IT보안 점검/원격지원 통화임을
  명시("IT 보안 점검·원격지원 상담 통화. 전문 용어를 그대로 전사하세요.")
- 적용 후 `scripts/bench_stt_matrix.py` + `scripts/stt_error_detector.py`로 WER/CER 회귀 측정.

## 5. Transcript correction loop 설계 개요

1. `stt_error_detector.py`로 오인식 후보 자동 추출(음운 유사·저신뢰·OOV).
2. 관리자 검수에서 confirmed/corrected 라벨 수집(기존 `pii_candidates` 검수 흐름 재사용 패턴).
3. 교정쌍을 `data/hotword_db.json`/도메인 사전에 승인 워크플로우로 반영(버전 관리).
4. 다음 재처리/신규 처리 시 hotwords·correction map 적용 → 루프.
- 본 단계는 설계만. 자동 교정 반영·재처리는 별도 승인.

## 6. 재처리 대상 선정 기준 (별도 승인 후 실행)

우선순위:
1. **short-heavy 세션** (짧은 단편 비율 ≥ 50%) — 체감 과분할 최대.
2. **납품 후보**(approved/both_agreed) 중 종결어미 비율 낮은 세션.
3. **pii_flag 세션** — 검수 품질 직접 영향.
- 재처리 시 v2 bidirectional + hotwords 동시 적용. 단계적 롤아웃(소수 세션 → 검수 → 확대).
